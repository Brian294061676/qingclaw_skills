# QingClaw Skills — 协作链路与行业套件方案

> 版本：v1.0 ｜ 最后更新：2026-07-03
> 对应规划：B4 协作链路强化（跨领域联动 / 数据契约 / 行业套件模板）

---

## 一、设计原则

1. **单一 Skill 可独立使用，链路组合产生增量价值**——链路是可选编排，不引入强依赖。
2. **数据经轻流应用中转**——Skill 之间不直接传递数据，上游 Skill 通过 `record_insert/record_update` 写入轻流应用，下游 Skill 通过 `record_list/record_get` 读取，轻流应用即为链路的"数据总线"。
3. **契约先行**——每条链路定义关键字段契约（字段名/类型/取值约定），上下游 Skill 的提示词中显式引用契约字段。
4. **人在环节点明确**——每条链路标注需要人工确认的关口（如排产确认、审批签字），AI 不越权。

## 二、核心跨领域链路（端到端）

### 链路 1：制造供应链闭环 ⭐ 旗舰链路

```
#57 需求预测 → #66 库存优化 → #52 智能排产 → #67 质量管控 → #53 交付预测 → #58 供应商履约
(SupplyChain)   (SupplyChain)   (Manufacturing)  (Manufacturing)  (Manufacturing)  (SupplyChain)
```

| 环节 | 上游产出（写入轻流） | 下游消费 |
|------|-------------------|---------|
| 需求预测 → 库存优化 | 分SKU预测量、置信区间 | 计算安全库存与补货量的需求输入 |
| 库存优化 → 智能排产 | 补货建议单（SKU/数量/时间窗口） | 转化为生产订单参与排产 |
| 智能排产 → 质量管控 | 排产计划（批次/产线/时间） | 按批次安排 IPQC 巡检计划 |
| 质量管控 → 交付预测 | 批次合格率、返工工时 | 作为延期风险因子输入 |
| 交付预测 → 供应商履约 | 高风险订单的物料缺口 | 触发对应供应商的交期催办 |

**人在环关口**：排产计划确认（生产计划员）、CAPA 措施审批（质量主管）。

### 链路 2：销售增长闭环

```
#16 线索挖掘 → #55 客户画像 → #17 跟进管理 → #51 录音分析 → #18 方案生成 → #20 赢输单分析
```

契约要点：线索表的 `意向评级`（A/B/C）驱动画像建档优先级；录音分析回写的 `客户意向等级` 与 `关键异议` 字段直接供方案生成引用；赢输单结论回流至线索评级规则优化。

### 链路 3：财务风控闭环

```
#10 发票处理 → #06 报销审核 → #07 财务分析 → #09 预算管控 → #56 回款预测 → #65 SOX审计
```

契约要点：发票台账的 `校验状态` 是报销审核的前置条件；预算执行偏差数据流入财务分析报告；审计助手以全链路留痕数据作为测试总体。

### 链路 4：客户之声闭环（VoC）

```
#63 智能客服 → #13 反馈分析 → #35 反馈跟踪 → #34 需求池 → #31 产品经理 → #63 闭环回访
```

契约要点：客服工单的 `问题分类` 与反馈应用打通；需求上线后经反馈跟踪应用自动通知原始工单客户，形成"提出→处理→告知"闭环。

### 链路 5：人才管理闭环

```
#01 招聘引擎 → #04 入职搭建 → #02 培训设计 → #03 绩效管理 → #68 薪酬分析 → #05 人力看板
```

契约要点：候选人评估数据入职后转为员工档案；绩效结果作为薪酬 CR 分析和调薪建议的输入；全链路指标汇聚人力看板。

### 链路 6：数据治理底座（横向支撑）

```
#64 数据质量检测 → 各领域数据型 Skill（#07/#11/#25/#57/#68 等）
```

任何分析类链路启动前，先由 #64 对数据源做质量体检并输出可信度评分，分析型 Skill 在结论中引用该评分作为置信度依据。

## 三、Skill 间数据契约标准

### 3.1 契约定义模板

每条链路在衔接处定义一份轻量契约（存于轻流应用字段配置 + Skill 提示词）：

```yaml
contract:
  name: 补货建议单            # 契约名称
  producer: 66-inventory-optimization
  consumer: 52-smart-production-scheduler
  carrier: 轻流应用「补货建议」 # 数据载体（轻流应用）
  fields:
    - {name: SKU编码,   type: text,   required: true}
    - {name: 建议数量,  type: number, required: true, rule: ">0"}
    - {name: 需求时间窗, type: date_range, required: true}
    - {name: 紧急程度,  type: select, options: [紧急, 高, 中, 低]}
    - {name: 置信度,    type: number, rule: "0-100"}
    - {name: 状态,      type: select, options: [待确认, 已确认, 已转排产, 已关闭]}
  handoff: 状态=已确认 时下游才可消费
```

### 3.2 通用约定

- **状态机字段必备**：每个契约载体应用必须有 `状态` 字段，下游只消费终态记录，避免消费中间数据。
- **来源回溯字段**：下游写入的记录携带 `来源记录ID`，支持全链路追溯。
- **置信度透传**：AI 生成的数据必须带 `置信度` 或 `AI生成` 标记，人工确认后才升级为业务事实。
- **幂等约定**：Skill 回写前先按业务键查重（如 SKU+日期），避免重复写入。

## 五、落地路线图

| 阶段 | 内容 | 验收标准 |
|------|------|---------|

- 本方案实现《QingClaw_Skills_项目介绍与规划》B4 全部三项内容。
